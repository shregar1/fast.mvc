## Configuration

FastMVC centralizes configuration in JSON files under `config/**/config.json`, paired with Pydantic DTOs and helper classes in `configurations/` and `dtos/configurations/`. Environment variables can override many of these settings.

This page summarizes the configuration model; see `configurations/README.md` and `LANDING.md` for deeper details.

### Configuration Sources

- **JSON config files**:
  - `config/db/config.json`
  - `config/cache/config.json`
  - `config/security/config.json`
  - Optional service configs like:
    - `config/mongo/config.json`
    - `config/cassandra/config.json`
    - `config/scylla/config.json`
    - `config/dynamo/config.json`
    - `config/cosmos/config.json`
    - `config/elasticsearch/config.json`
    - `config/email/config.json`
    - `config/slack/config.json`
    - `config/datadog/config.json`
    - `config/telemetry/config.json`
    - `config/payments/config.json`
    - `config/identity/config.json`

- **DTOs (Pydantic models)**:
  - Under `dtos/configurations/` (for example `dtos/configurations/datastores.py`, `dtos/configurations/communications.py`, etc.).
  - Group related configs (e.g. `DatastoresConfigurationDTO`, `CommunicationsConfigurationDTO`, `PaymentsConfigurationDTO`).

- **Configuration managers**:
  - Under `configurations/` (for example `configurations/db.py`, `configurations/cache.py`, `configurations/security.py`, `configurations/identity.py`).
  - Encapsulate loading, caching, and environment-variable overrides.

- **Environment variables**:
  - `.env` / `.env.example` generated by `ProjectGenerator` (`_create_env_example`).
  - Used for runtime overrides and secrets (DB passwords, JWT secrets, API keys).

### Core Configurations

#### Database (`config/db/config.json`, `configurations/db.py`)

Controls your primary relational database:

- User, password, host, port, database name.
- SQLAlchemy connection string (driver varies for Postgres/MySQL/SQLite).
- Pruned and customized by the generator based on wizard choices.

Usage:

```python
from configurations.db import DBConfiguration

db_config = DBConfiguration()
dto = db_config.get_config()
connection_string = dto.connection_string
```

#### Cache / Redis (`config/cache/config.json`, `configurations/cache.py`)

Configures Redis for:

- Caching.
- Rate limiting.
- Session-like state.

Usage:

```python
from configurations.cache import CacheConfiguration

cache_cfg = CacheConfiguration().get_config()
```

Redis itself can also be toggled via `.env` (`REDIS_*` variables).

#### Security (`config/security/config.json`, `configurations/security.py`)

Holds:

- Security headers (CSP, HSTS, etc.).
- Input validation rules (max string length, minimum password length).
- Authentication settings (JWT expiry, max login attempts).

Supports environment variable overrides such as:

- `SECURITY_HSTS_MAX_AGE`
- `SECURITY_ENABLE_CSP`
- `SECURITY_MAX_STRING_LENGTH`
- `SECURITY_JWT_EXPIRY_MINUTES`

See `configurations/README.md` for the full mapping table.

### Optional Integrations

FastMVC can include or later add optional services. The presence of their configs and DTOs depends on:

- CLI flags at generate time (see `LANDING.md` and [`cli.md`](cli.md)).
- `fastmvc add service <name>` for existing projects.

Examples:

- **Datastores** (`dtos/configurations/datastores.py`):
  - `DBConfigurationDTO`, `CacheConfigurationDTO`, `MongoConfigurationDTO`, `CassandraConfigurationDTO`, `ScyllaConfigurationDTO`, `DynamoConfigurationDTO`, `CosmosConfigurationDTO`, `ElasticsearchConfigurationDTO`.
- **Communications** (`dtos/configurations/communications.py`):
  - `EmailConfigurationDTO`, `SlackConfigurationDTO`, `PushConfigurationDTO`.
- **Observability**:
  - Datadog (`config/datadog/config.json`, `DatadogConfigurationDTO`).
  - OpenTelemetry (`config/telemetry/config.json`, `TelemetryConfigurationDTO`).
- **Payments**:
  - `config/payments/config.json`, DTOs for Stripe/Razorpay/PayPal/PayU/Link.
- **Identity / SSO**:
  - `config/identity/config.json`, DTOs in `dtos/configurations/identity/`, services in `services/auth/`.

These are designed so your UI or scripts can safely toggle them using the CLI flags listed in `LANDING.md`.

### Environment Variables and `.env`

The generator creates `.env.example` with sensible defaults derived from your wizard choices (`db_backend`, `use_redis`, etc.). You typically:

```bash
cp .env.example .env
```

Key sections inside `.env`:

- **Application**: app name, environment, host/port.
- **Database**: host, port, name, user, password, pool settings.
- **Redis**: host/port/password, or commented out if disabled.
- **Optional datastores**: toggles and connection strings for MongoDB, Cassandra, Scylla, Dynamo, Cosmos, Elasticsearch.
- **Email & Slack**: SMTP/SendGrid and Slack webhook/bot tokens.
- **Datadog & OpenTelemetry**: enable flags and exporter endpoints.
- **Payments**: `PAYMENTS_ENABLED`, default provider.
- **Security**: `JWT_SECRET_KEY`, `BCRYPT_SALT`, CORS settings.
- **Rate limiting & logging**.

This file is meant to be customized per environment and should usually **not** be committed (only `.env.example` is tracked).

### Best Practices

- **Keep secrets out of Git**:
  - Use `.env` and environment variables, not JSON configs, for credentials.
- **Use config DTOs in code**:
  - Import from `configurations.*` and `dtos.configurations.*` instead of reading JSON directly.
- **Leverage grouped DTOs**:
  - For example, use `DatastoresConfigurationDTO` to pass all datastore settings into a service or initializer.
- **Let the generator prune unused configs**:
  - Rely on CLI flags and `fastmvc add service` so your projects stay lean.

For a feature-by-feature matrix of which configs and DTOs are involved, see `LANDING.md`.
